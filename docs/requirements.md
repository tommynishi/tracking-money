# 要件定義書（requirements.md）

Tracking Money の要件定義書です。

本書はプロジェクトにおける最優先ドキュメントであり、他のドキュメント（architecture.md / database.md / api.md / screen.md 等）と仕様が競合する場合は本書を正とします。

---

# 1. プロジェクト概要

## 1.1 目的

Tracking Money は、AIを活用した家計簿・資産管理Webアプリです。

カード会社から取得したCSV・PDF明細を取り込み、個人・家族単位で支出を管理し、AIによる分析・分類で家計の把握と改善を支援します。

## 1.2 ユーザーの最初の成功体験

「カード会社からダウンロードした明細ファイルを取り込むだけで、今月なにに・いくら使ったかがカテゴリ別に一目で分かる」

## 1.3 開発方針

* MVPではなく、長期運用を前提としたプロダクトとして開発する
* 品質・保守性・拡張性・可読性を最優先とする
* 将来機能（銀行連携・資産管理等）を見据えた拡張しやすい設計を維持する

# 2. 利用者・利用形態

| 項目 | 内容 |
| --- | --- |
| 利用形態 | 開発者本人とその家族のみで利用する（一般公開しない） |
| 想定ユーザー数 | 数名程度（1〜2世帯） |
| ユーザー登録 | LINE Loginによる認証。一般公開はしないが、認証・認可は省略しない |
| 利用環境 | Windows / macOS / Android / iPhone / タブレット（レスポンシブ対応） |

一般公開を前提としないため、マルチテナントSaaSとしてのスケール要件・利用規約・課金は本要件のスコープ外とします。ただし、認証・認可・データ分離は限定利用であっても必須とします。

# 3. 用語定義

| 用語 | 定義 |
| --- | --- |
| 家計簿（帳簿） | 明細を記録する単位。「個人家計簿」と「家族家計簿」の2種類がある |
| 個人家計簿 | ユーザー本人のみが参照・更新できる家計簿 |
| 家族家計簿 | 作成者に招待されたユーザーも参照・更新できる共有家計簿 |
| 明細 | 1件の支出記録（利用日・金額・摘要・カテゴリ等） |
| カテゴリ | 明細の分類（食費・日用品・固定費など）。ユーザーが編集可能 |
| インポート | CSV・PDFファイルから明細を一括登録する機能 |
| 重複チェック | 取込済み・登録済みの明細と同一と思われる明細を検知する機能 |

# 5. 機能要件

要件IDは `FR-<領域>-<連番>` で表記します。

## 5.1 認証・アカウント（AUTH）

| ID | 要件 |
| --- | --- |
| FR-AUTH-01 | ユーザーはLINE Loginでログインできる |
| FR-AUTH-02 | 未認証ユーザーはログイン画面以外へアクセスできない |
| FR-AUTH-03 | 初回ログイン時にユーザーアカウントを自動作成し、LINEの表示名・アイコンを初期プロフィールとする |
| FR-AUTH-04 | ユーザーは表示名を変更できる |
| FR-AUTH-05 | ログアウトできる |

## 5.2 家計簿（帳簿）管理（LEDGER）

| ID | 要件 |
| --- | --- |
| FR-LEDGER-01 | ユーザーは個人家計簿を1つ作成できる |
| FR-LEDGER-02 | ユーザーは家族家計簿を1つ作成できる |
| FR-LEDGER-03 | 個人家計簿は本人以外のユーザーから参照・更新できない |
| FR-LEDGER-04 | 家族家計簿は、招待され参加したユーザーも参照・更新できる |
| FR-LEDGER-05 | ユーザーが同時に紐づける（作成または参加できる）家族家計簿は最大1つとする |
| FR-LEDGER-06 | ユーザーは操作対象の家計簿（個人／家族）を切り替えられる |
| FR-LEDGER-07 | 家計簿の名称を変更できる |
| FR-LEDGER-08 | 家計簿の削除は論理削除とし、削除前に確認を求める |

## 5.3 家族招待・共有（INVITE）

| ID | 要件 |
| --- | --- |
| FR-INVITE-01 | 家族家計簿の作成者は、登録済みユーザーをアプリ内で検索・指定して招待できる |
| FR-INVITE-02 | 招待されたユーザーには招待通知（アプリ内通知）が届き、承諾または拒否を選択できる |
| FR-INVITE-03 | 招待されたユーザーが既に自分の家族家計簿を持っている場合、「自分の家族家計簿を削除して参加する」か「招待を拒否する」かを選択させる |
| FR-INVITE-04 | 招待に参加したユーザーは、新たに家族家計簿を作成できない |
| FR-INVITE-05 | 家族家計簿の作成者は、参加メンバーを一覧・除外できる |
| FR-INVITE-06 | 参加メンバーは自ら家族家計簿から退出できる |

## 5.4 明細管理（ENTRY）

対象は**支出のみ**とします（収入・資産はPhase 4の資産管理で扱う）。

| ID | 要件 |
| --- | --- |
| FR-ENTRY-01 | 明細を手入力で登録できる（利用日・金額・摘要・カテゴリ・支払方法・メモ） |
| FR-ENTRY-02 | 明細を編集できる |
| FR-ENTRY-03 | 明細を削除できる（論理削除） |
| FR-ENTRY-04 | 明細を一覧表示できる（月単位を基本とし、期間・カテゴリ・金額・キーワードで絞り込み、日付・金額でソート、ページング対応） |
| FR-ENTRY-05 | 明細には登録者（どのユーザーが登録したか）を記録し、家族家計簿では登録者を表示する |
| FR-ENTRY-06 | 明細には取込元（手入力／CSV／PDF）を記録する |
| FR-ENTRY-07 | 金額は日本円（整数）のみ扱う |

## 5.5 カテゴリ管理（CATEGORY）

| ID | 要件 |
| --- | --- |
| FR-CATEGORY-01 | 家計簿ごとにカテゴリを管理できる（追加・名称変更・削除・並び替え） |
| FR-CATEGORY-02 | 家計簿作成時にデフォルトカテゴリ一式を自動投入する（例：食費・日用品・交通費・住居・水道光熱費・通信費・保険・医療・教育・娯楽・被服・交際費・サブスク・その他） |
| FR-CATEGORY-03 | カテゴリの削除は、使用中の明細がある場合「その他」等への付け替えを行ってから削除する（明細のカテゴリ欠損を発生させない） |
| FR-CATEGORY-04 | カテゴリには「固定費」フラグを設定でき、AI分析（固定費分析）で利用する |

## 5.6 CSVインポート（CSV）

| ID | 要件 |
| --- | --- |
| FR-CSV-01 | 以下のカード会社のCSV明細を取り込める：楽天カード（楽天e-NAVI）／JCBカード（MyJCB）／Eposカード／セゾンカード |
| FR-CSV-02 | 上記以外のフォーマットは汎用（列マッピング）方式で取り込める。取込時にユーザーが「利用日・金額・摘要」等の列を対応付け、そのマッピングを保存・再利用できる |
| FR-CSV-03 | フォーマット（カード会社）はファイル内容から自動判定を試み、判定できない場合はユーザーが選択する |
| FR-CSV-04 | 取込前にプレビューを表示し、ユーザーが内容（カテゴリ・除外行）を確認・修正してから確定できる |
| FR-CSV-05 | 取込結果（成功件数・スキップ件数・エラー件数）を表示する |
| FR-CSV-06 | 文字コード（UTF-8／Shift_JIS）を自動判定する |
| FR-CSV-07 | 一部の行が不正でも取込全体を失敗させず、不正行はエラー内容とともに報告する |

## 5.7 PDF（OCR）インポート（PDF）

対象は**カード会社が発行する明細PDFのみ**とします（レシート撮影は将来機能）。

| ID | 要件 |
| --- | --- |
| FR-PDF-01 | カード明細PDFをアップロードし、OpenAI Visionで明細（利用日・金額・摘要）を抽出できる |
| FR-PDF-02 | 抽出結果は必ずプレビューで表示し、ユーザーが確認・修正してから登録する（OCR結果を無確認で登録しない） |
| FR-PDF-03 | 抽出に失敗した場合、失敗した旨と対処（CSVでの取込・手入力）を案内する。AI失敗によりアプリが異常動作しないこと |
| FR-PDF-04 | 対応カード会社はCSVと同一（楽天・JCB・Epos・セゾン）を優先対応とする |

## 5.8 重複チェック（DUP）

| ID | 要件 |
| --- | --- |
| FR-DUP-01 | インポート時、同一家計簿内に「利用日・金額・摘要（正規化後）」が一致する明細が既に存在する場合、重複候補として警告する |
| FR-DUP-02 | 重複候補はユーザーが1件ずつ、または一括で「取り込む／スキップする」を選択できる（既定はスキップ） |
| FR-DUP-03 | 同一ファイル（内容ハッシュが一致）を再度取り込もうとした場合、取込済みである旨を警告する |
| FR-DUP-04 | 重複チェックは自動削除を行わない（ユーザーの確認なくデータを消さない） |

## 5.9 Google Drive連携（DRIVE）

保存先は**アプリが管理する共通のGoogle Drive**（サービスアカウント等でアプリが保持するDrive）とします。ユーザー個人のDrive連携は行いません。

| ID | 要件 |
| --- | --- |
| FR-DRIVE-01 | インポートしたCSV・PDFの原本ファイルを共通Driveへ保存する |
| FR-DRIVE-02 | Drive上のフォルダは家計簿単位で分離する |
| FR-DRIVE-03 | 取込履歴からファイルをダウンロードできる（アプリの認可チェックを経由し、本人がアクセス権を持つ家計簿のファイルのみ） |
| FR-DRIVE-04 | 取込履歴からファイルを削除できる |
| FR-DRIVE-05 | 取込履歴にDriveファイルへのリンクを表示する |
| FR-DRIVE-06 | Driveへの保存に失敗しても明細の取込自体は成功させ、ファイル未保存である旨を記録・表示する |

## 5.10 LINE通知（NOTIFY）

| ID | 要件 |
| --- | --- |
| FR-NOTIFY-01 | 月次定期リマインド：毎月ユーザーが設定した日に「明細を取り込みましょう」と通知する |
| FR-NOTIFY-02 | 未登録検知リマインド：一定期間（既定7日・ユーザー設定可）明細の登録・インポートがない場合に通知する |
| FR-NOTIFY-03 | ユーザーは各通知を個別にON/OFFできる |
| FR-NOTIFY-04 | 通知の送信失敗はログへ記録し、アプリ本体の動作へ影響させない |

## 5.11 AI分析（AI）

| ID | 要件 |
| --- | --- |
| FR-AI-01 | 今月支出分析：今月の支出をカテゴリ別に集計・可視化し、AIによる所見を表示する |
| FR-AI-02 | 前月比較：前月との増減をカテゴリ別に比較する |
| FR-AI-03 | 前年同月比較：前年同月との増減を比較する |
| FR-AI-04 | カテゴリ別推移：カテゴリごとの月次推移をグラフ表示する |
| FR-AI-05 | 支出ランキング：期間内の支出を金額順に表示する |
| FR-AI-06 | 固定費分析：固定費カテゴリの支出を集計・分析する |
| FR-AI-07 | サブスク検知：毎月同額・同摘要の支出をサブスクリプション候補として検知する |
| FR-AI-08 | 節約提案：支出傾向をもとにAIが節約案を提示する |
| FR-AI-09 | 来月支出予測：過去実績をもとに来月の支出を予測する |
| FR-AI-10 | AI出力は構造化データ（JSON）として受け取り、画面表示に利用する |
| FR-AI-11 | AI呼び出しが失敗しても、集計・グラフ等の非AI機能は正常に動作すること |

## 5.12 AIカテゴリ自動分類（AICAT）

| ID | 要件 |
| --- | --- |
| FR-AICAT-01 | インポート時、明細の摘要からAIがカテゴリを自動判定し、初期値として設定する |
| FR-AICAT-02 | AI判定結果はユーザーがプレビュー・明細編集で変更できる |
| FR-AICAT-03 | ユーザーが修正したカテゴリ（摘要とカテゴリの対応）を記憶し、同じ摘要は次回以降その対応を優先する |
| FR-AICAT-04 | AI判定に失敗した明細は「未分類（その他）」として取り込み、取込自体は成功させる |

## 5.13 ダッシュボード（DASH）

| ID | 要件 |
| --- | --- |
| FR-DASH-01 | ログイン後のトップ画面として、今月の支出合計・カテゴリ別内訳・直近明細・前月比を表示する |
| FR-DASH-02 | 個人家計簿・家族家計簿を切り替えて表示できる |

# 6. 非機能要件

要件IDは `NFR-<連番>` で表記します。

| ID | 分類 | 要件 |
| --- | --- | --- |
| NFR-01 | 性能 | 主要画面（ダッシュボード・明細一覧）は通常時3秒以内に表示する |
| NFR-02 | 性能 | 明細一覧はページング必須とし、10年分・数十万件規模のデータでも実用速度を維持する（適切なIndex・N+1回避） |
| NFR-03 | セキュリティ | 全APIで認証を必須とし、家計簿単位の認可チェック（本人・参加メンバーのみ）を行う。Supabase RLSを併用する |
| NFR-04 | セキュリティ | SQL Injection・XSS・CSRF対策を行う |
| NFR-05 | セキュリティ | 個人情報（明細内容・LINE ID等）をログへ出力しない |
| NFR-06 | セキュリティ | APIキー・シークレットは環境変数で管理し、リポジトリへコミットしない |
| NFR-07 | UI | レスポンシブ対応。PCは表形式、スマートフォンはカード形式を基本とする |
| NFR-08 | UI | ダークモード対応、アクセシビリティ考慮 |
| NFR-09 | 可用性 | Vercel・Supabaseのマネージドサービスを利用する。DBはSupabaseのバックアップ機能で復旧可能とする |
| NFR-10 | 保守性 | TypeScript strict・any禁止・ESLintエラー0を維持する（詳細は coding-rules.md） |
| NFR-11 | 保守性 | Feature First Architecture・疎結合設計を維持する（詳細は architecture.md） |
| NFR-12 | テスト | Unit Test・Integration Testを作成する。必要に応じてE2E Test |
| NFR-13 | コスト | OpenAI API・LINE APIの呼び出しは必要最小限とし、分析結果は再利用（キャッシュ）を検討する |
| NFR-14 | データ | 主要テーブルはUUID主キー・created_at・updated_at・論理削除・外部キー制約を備える（詳細は database.md） |

# 8. 想定される失敗ケースと設計方針

| 失敗ケース | 方針 |
| --- | --- |
| CSVフォーマットがカード会社の仕様変更で変わる | フォーマット定義をパーサーとして分離し、会社ごとに独立して修正できる構造にする。汎用（列マッピング）方式をフォールバックとする |
| OCR（OpenAI Vision）の抽出精度が不十分 | 必ずプレビュー確認を挟む。失敗時はCSV取込・手入力へ誘導する |
| AI APIの障害・タイムアウト | 非AI機能（記録・集計・グラフ）はAIなしで完結させる。AI部分のみ「取得できませんでした」と表示 |
| 重複取込によるデータ汚染 | 重複チェック（明細単位＋ファイル単位）と取込前プレビューで防止。自動削除はしない |
| Drive保存失敗 | 明細取込は成功させ、ファイル未保存を記録。後から再アップロード可能とする |
| 家族家計簿の招待事故（誤招待） | 招待は承諾制。作成者によるメンバー除外・本人退出を可能とする |
| 誤削除 | 論理削除＋削除前確認 |
