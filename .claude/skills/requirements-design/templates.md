# ドキュメントテンプレート

docs/ 配下の各ドキュメントの節構成。新規節の追加・既存節の更新はこの構成に合わせる。
全ドキュメント共通: 末尾に「# 改訂履歴」（日付 / 版 / 変更内容の表）を持つ。

## requirements.md（要件定義書）

```markdown
# 1. プロジェクト概要
## 1.1 目的
## 1.2 ユーザーの最初の成功体験   ← 1行で定義
## 1.3 開発方針
# 2. 利用者・利用形態
# 3. 用語定義                    ← 表形式。新しい用語は必ずここに追加
# 4. スコープとフェーズ計画       ← Phase 1〜4
# 5. 機能要件                    ← 領域ごとに小節。各要件は FR-<領域>-<連番>
## 5.x <機能名>（<領域コード>）
# 6. 非機能要件
# 7. 前提条件・制約事項
# 8. 想定される失敗ケースと設計方針
# 9. スコープ外（本要件では実装しない）
# 10. 未確定事項（実装前に確定が必要）
# 11. 完成条件（Definition of Done）
```

機能要件の書式:

```markdown
| ID | 要件 | 優先度 |
| --- | --- | --- |
| FR-ENTRY-01 | ユーザーは明細を登録できる | Must |
```

## architecture.md（システム構成書）

```markdown
# 1. システム全体構成           ← Mermaid構成図 + 構成要素と役割
# 2. 設計原則
# 3. 主要な設計判断（3案比較）   ← 新しい重要判断はここに小節を追加
## 3.x <判断テーマ>             ← 案A/B/C、採用案、採用理由・不採用理由
# 4. レイヤー構成
# 5. ディレクトリ構成
# 6. 認証・認可フロー
# 7. 主要データフロー            ← 機能ごとにシーケンス/フロー図
# 8. 外部サービス連携方針
# 9. 環境構成
# 10. 横断的関心事（エラーハンドリング / パフォーマンス / テスト戦略）
# 11. 拡張性の考慮
```

## database.md（テーブル定義書）

```markdown
# 1. 共通ルール
## 1.1 共通カラム   ← id UUID PK / created_at / updated_at / deleted_at（論理削除）
## 1.2 設計規約     ← snake_case、外部キー制約、適切なIndex
## 1.3 RLS（Row Level Security）
# 2. ER図           ← Mermaid erDiagram。テーブル追加時は必ず更新
# 3. テーブル定義
## 3.x <table_name>（<和名>）
```

テーブル定義の書式:

```markdown
| カラム | 型 | NULL | デフォルト | 説明 |
| --- | --- | --- | --- | --- |

Index / 制約 / RLSポリシー を続けて記載
```

## api.md（API仕様書）

```markdown
# 1. 共通仕様       ← 基本 / HTTPステータス / レスポンス形式 / ページング・フィルタ・ソート
# 2〜n. <リソース>（<領域コード>）
## n.x <METHOD> /api/...
```

エンドポイントの書式:

- 対応する要件ID（FR-XXX-nn）を明記
- リクエスト（パスパラメータ / クエリ / ボディ）とレスポンス例（JSON）
- エラー時のHTTPステータス
- 一覧APIは共通のページング・フィルタ・ソート仕様に従う

## screen.md（画面設計書）

```markdown
# 1. 画面一覧       ← SCR-nn / 画面名 / パス / 対応要件ID の表
# 2. 画面遷移図     ← Mermaid
# 3. 共通レイアウト
# 4. 画面詳細
## SCR-nn <画面名>  ← 目的 / 表示項目 / 操作 / 使用API / 状態（ローディング・空・エラー）
# 5. アクセシビリティ・レスポンシブ指針
```

画面詳細の必須観点:

- PCは表形式、スマートフォンはカード形式
- ローディング・空状態・エラー状態の表示
- 使用するAPIエンドポイント（api.md と一致させる）

## ID採番規則

- 機能要件: `FR-<領域コード>-<2桁連番>`（例: FR-ENTRY-01）
- 画面: `SCR-<2桁連番>`（エラー等の特殊画面は SCR-90番台）
- 既存の最大番号の次から採番する。欠番の再利用・意味変更は禁止
